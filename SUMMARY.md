# Memory Decay 项目总结

## 时间线

**2026-03-18 下午**，从思辨到落地，6小时完整实现。

## 核心问题

Peon 做产品落地页时，检索到一个废弃的试验目录路径（`/tmp/landing-test/`），当成有效信息去执行，浪费了大量 token。

**根本原因：** 当前记忆系统是「图书馆模型」——所有记忆平权，没有时间衰减，没有可信度区分。三个月前的记忆和昨天的记忆，检索时置信度一样。

## 解决方案

模拟人类模糊记忆：**写入打标 + 时间梯度衰减 + 混合检索**

### 1. 写入打标（Birth Tag）

```markdown
<!-- meta: type=decision, ttl=permanent, confidence=0.9 -->
选择 Creem 作为支付平台，因为国内友好。
```

- `type`: decision / experiment / reference / status / temporary
- `ttl`: 3d / 7d / 30d / permanent
- `confidence`: 0.0-1.0

### 2. 时间梯度衰减

| 年龄 | 层级 | 保留内容 |
|------|------|----------|
| 0-3天 | fresh | 完整原文 |
| 4-14天 | recent | 原文 |
| 15-30天 | faded | 仅摘要 |
| 30天+ | ghost | 一行索引 |
| ttl过期 | expired | 检索时跳过 |

### 3. 混合检索

- **关键词**：TF-IDF + domain 别名（"收费"→payment）
- **语义**：Qwen3-Embedding-8B（中文理解）
- **融合**：kw 40% + sem 60%

## 技术实现

- **语言**：Node.js (纯 ESM)
- **存储**：文件系统 JSON
- **分词**：CJK bigram + 英文单词
- **embedding**：Qwen3-Embedding-8B via 302.ai
- **代码量**：~600 行（不含测试）

## 测试结果

### 数据集演变

1. **初始**：124条（episodic + semantic + procedural）
   - 问题：61% 是垃圾（Session Key、对话碎片）
2. **过滤后**：109条（去重）
   - 问题：还有大量对话记录噪音
3. **最终**：24条（仅 semantic + procedural）
   - 质量：可用，但仍有元信息混入

### 检索准确率

**精确查询（100%）：**
- "peonai 做了哪些产品" → top-3 全命中 ✅
- "插件上架了吗" → top-1 精准 ✅

**模糊查询（50%）：**
- "好像之前搞过收费的事" → payment 排第2（被"我是 Peon"压了）
- "那个收费的事" → 同上

**失败案例：**
- "之前踩过一个坑" → 完全没命中 bug 记录

## 核心发现

### ✅ 架构是对的

1. 时间衰减机制工作正常（fresh→faded→expired）
2. 过期过滤 100% 可靠（0 泄漏）
3. 混合检索优于单一方法
4. 千问 embedding 中文理解能力强

### ❌ 瓶颈是数据质量

**问题根源：**

episodic 文件本质是 session transcript（对话记录），不是结构化记忆。从对话里抽第一行当 summary，信息密度太低。

**坏 summary 示例：**
- "我是 Peon 🔨，悦哥你的 AI 助手"（自我介绍）
- "好，继续~接近目标了"（对话碎片）
- "Session Key: agent:main:..."（元数据）

**好 summary 示例：**
- "Creem 支付集成：选择 Creem 而非 Stripe，因为国内友好"
- "博客双语方案：中文 .md + 英文 .en.md，Hugo i18n 原生支持"

### 🎯 关键洞察

**模糊记忆的本质不是模糊检索，是信息压缩。**

人类的模糊记忆：
- 具体：`修了 OpenClaw 飞书插件 bug：opus 用 msg_type:"media" 改为 "audio"`
- 模糊：`飞书语音消息格式问题`
- 极度模糊：`飞书出过问题`

当前系统只做了时间衰减，没做**语义压缩**——faded 层应该是摘要，ghost 层应该是索引，但现在都还是原文。

## 能否无痛使用？

**前提条件：**

✅ 数据质量高（结构化陈述句，不是对话记录）  
✅ 严格打标（type/ttl/confidence）  
✅ 定期维护（自动 decay）  
✅ 检索习惯（模糊用 hybrid，精确用 search）

**结论：**

如果 agent 严格遵守写入规范，**可以无痛使用**。

但当前 OpenClaw 的 memory 文件（包括 semantic/procedural）混杂了太多元信息、流程说明、自我介绍，不适合直接导入。
